**数据流向：**
- 固件每 1 秒通过 HTTP POST 将 10 帧数据上传到 `dataIngest` 云函数，存入云数据库
- 小程序通过云函数从云数据库读取历史数据
- 配网：小程序通过 BLE 将 WiFi 凭证发送给固件
- 命令：云端通过 `getDeviceCommand` 云函数下发指令，固件轮询获取

## 2. 固件端详解

### 2.1 入口: main.cpp

**setup() 执行顺序：**

```
1. delay(3000)                  ← 等待串口监控连接
2. Serial.begin(115200)        ← 调试串口
3. analogReadResolution(14)    ← 14-bit ADC 精度
4. gSignal.init()              ← 环形缓冲、FFT 旋转因子
5. 1kHz FspTimer 初始化         ← 硬件定时器，timer_callback ISR
6. gState.init()               ← ST_BOOT
7. gStorage.Init()             ← EEPROM 初始化 + 校准数据读取
8. gNetManager.init()          ← WiFi连接 + NTP同步 + 云端上传启动
9. gBleConfig.init()           ← BLE 广播
10. gAppController.init()      ← 加载校准参数 → ST_RUNNING
11. 注册 JSON 命令回调
```

**loop() 主循环（10Hz, LOOP_INTERVAL_MS=100）：**

```
if g_adcTimerFlag:
  g_adcTimerFlag = false
  raw = analogRead(A0); analogRead(A0)  ← FAST_ADC_READ 宏（双读清管道）
  gSignal.isrPushSample(raw)            ← 推入512深环形缓冲
  gSignal.updateSampleRateStats()       ← 采样率统计

if millis() - lastTick < 100ms: return  ← 10Hz 节拍控制

gStorage.tick()              ← 空函数（云方案无本地存储轮询）
gNetManager.tick()           ← HTTP上传 + 云端命令轮询 + WiFi 状态管理
gBleConfig.tick()            ← BLE 连接管理
gProtocol.tick()             ← 延迟 JSON 命令处理（中断安全）
cmd = gProtocol.tickLocalDebug()  ← 串口调试命令解析
gAppController.tick()        ← 核心业务调度
```

### 2.2 信号处理: SignalProcessor

> 📖 详细算法说明（RMS/MDF/Activation/Fatigue/Quality 的完整计算公式与设计原理）请参见 [算法说明文档](./算法说明文档.md)。

**核心数据流（每 100ms 一次 update() 调用）：**

```
1. drainNewSamples() → 从环形缓冲提取新样本
2. takeSnapshotIfNeeded(256) → 从环形缓冲拷贝256点到快照缓存
   - DC偏置：简单均值估计
3. calculateRMS()
   - 256点 AC 耦合（减 DC）
   - RMS = sqrt(Σ(AC²)/256)
   - 输出单位：mV（通过 m_mvPerAdcUnit 转换）
4. calculateMDF()
   - takeSnapshot → calculatePowerSpectrum()
   - Hanning 窗加权
   - 256点 FFT（Cooley-Tukey 自实现，bitReverse + 迭代蝶形）
   - 功率谱 = (real²+imag²)/N
   - findMedianFrequency(10~250Hz)：累积功率过半处
   - EMA 平滑（α 自适应 0.15~0.35）
5. updateFatigue(rms, mdf)
   - 激活度：A = (rms-relax_rms)/(active_rms-relax_rms)，clamp [0,1]
   - 收缩检测：rms > relax_rms×2
   - 疲劳度：F = (baselineMDF-mdf)/baselineMDF×100，EMA(α=0.1)
   - 动态基线：activation>20% 时捕获当前MDF为baseline
```

**EMA 策略（MDF 平滑）：**

| 场景 | α 值 | 说明 |
|------|------|------|
| 肌肉收缩中 | 0.35 | 快速跟踪频谱变化 |
| MDF 下降（疲劳趋势） | 0.35 | 加速响应 |
| 稳态连续帧 ≥10 | 0.15 | 平滑降噪 |
| 稳态未满10帧 | 0.5→0.15 渐变 | 快速初始化后稳定 |

**MDF 有效性判断：**
- rawMDF 在 [10, 250] Hz 范围内 → 生理有效
- rawMDF 在 [8, 10) Hz → 可接受但质量较低
- 异常帧：保持上次有效值，不更新 EMA

**校准 MDF 缓冲（v3.9.24）：**
- ACTIVE 阶段每 100ms 记录一次 MDF 到 200 深环形缓冲
- finalizeCalibMdf() 计算峰值和末尾 5 点去极值均值

### 2.4 EEPROM 存储: StorageManager

**存储介质：** RA4M1 板载 Data Flash（8KB），通过 Arduino EEPROM 库访问。
云方案已移除 SPI Flash 及 A/B/C 分区架构，实时数据通过 WiFi 直接上传云端。

**EEPROM 地址布局：**

```
0x00 ──────── 32B ── WiFi SSID
0x20 ──────── 64B ── WiFi Password
0x60 ──────── 1B  ── WiFi Magic (0xA5)
0x70 ──────── 28B ── 校准数据 (PersonalCalibData_t)
0x90 ──────── 35B ── 用户信息 (UserProfileData_t)
0xB4 ──────── 1B  ── 校准 Magic (0xAA)
0xB5 ──────── 1B  ── Profile Magic (0xBB)
```

**校准数据结构（28 字节）：**

```c
PersonalCalibData_t {
  relax_rms_mv:      float    // 放松 RMS (mV)
  active_rms_mv:     float    // 收缩 RMS (mV)
  calib_timestamp_sec: uint32 // 校准时间（秒）
  calib_timestamp_ms: uint16  // 校准时间（毫秒）
  relax_mdf_hz:      float    // 放松 MDF (Hz)
  active_mdf_hz:     float    // 收缩 MDF 峰值 (Hz)
  end_mdf_hz:        float    // 收缩末尾 MDF (Hz)
}
```

**用户信息结构（35 字节）：**

```c
UserProfileData_t {
  name[32]:    char     // 姓名
  age:         uint8    // 年龄
  gender:      uint8    // 1:男, 2:女
  handedness:  uint8    // 1:左手腕, 2:右手腕
}
```

**数据权威性：** EEPROM 为唯一权威数据源，小程序 localStorage 仅作快照缓存用于快速 UI 展示。

**校准数据验证：**
- Magic 字节检查（0xAA）
- NaN 检测（所有 float 字段合理性检查）

**关键 API：**

| 函数 | 说明 |
|------|------|
| `Init()` | 初始化 EEPROM，打印校准有效状态 |
| `GetPersonalCalib(data)` | 读取校准数据（含 NaN 校验） |
| `UpdatePersonalCalib(data)` | 保存校准数据（含 NaN 防御） |
| `GetUserProfile(profile)` | 读取用户信息 |
| `SetUserProfile(profile)` | 保存用户信息 |
| `LoadWifiCredentials(creds)` | 读取 WiFi 凭证 |
| `SaveWifiCredentials(creds)` | 保存 WiFi 凭证 |
| `tick()` | 空函数（云方案无本地存储轮询） |

**EEPROM 特性：**
- 总容量：8 KB
- 写入方式：EEPROM.update()（仅变化时写入，减少磨损）
- 擦写寿命：> 100,000 次

### 2.5 网络通信: NetManager（云端 HTTP 上传方案）

**架构：** 固件通过 HTTP POST 将数据上传到微信云开发云函数，不再使用 WebSocket 直连小程序。

```
固件 ──HTTP POST──► 微信云函数(dataIngest) ──► 云数据库
小程序 ──云函数调用──► 云数据库 ← 读取固件上传的数据
```

**上传格式（v3.9.45 简化后）：**

```json
{
  "points": [
    [852, 800, 849, 223, 85],
    [860, 810, 845, 225, 86],
    ...
  ]
}
```

字段顺序固定：`[rms×1000, activation×10, mdf×10, fatigue×10, quality]`，不需要传 `session_id`、`device_id`、`fields`。

**上传参数（v3.9.45）：**

| 常量 | 值 | 说明 |
|------|-----|----------|
| `INGEST_BATCH_FRAMES` | 10 | 每批上传帧数（1秒 @ 10Hz） |
| `INGEST_RETRY_QUEUE` | 150 | 断网重试队列最大帧数（15秒缓存） |
| 上传频率 | 1秒/次（或缓冲区满） | 实时性约 1 秒延迟 |

**上传触发条件：**
- 时间触发：距上次上传 ≥ 1秒
- 缓冲区满：`_batchCount >= INGEST_BATCH_FRAMES`（10帧）

**断网重试机制：**
- `pushDataPoint()` 先将数据存入 `_retryQueue`（150帧环形缓冲）
- 网络恢复后，下次成功上传时会将重试队列中的数据补传到云端
- 队列满时覆盖最旧数据（环形覆盖）

**云端通信链路：**

| 功能 | 云函数 | HTTP 路径 |
|------|--------|----------|
| 设备注册 | `deviceRegister` | `/deviceRegister` |
| 数据上传 | `dataIngest` | `/dataIngest` |
| 命令拉取 | `getDeviceCommand` | `/getDeviceCommand` |
| 状态上报 | `reportDeviceStatus` | `/reportDeviceStatus` |
| 命令确认 | `ackDeviceCommand` | `/ackDeviceCommand` |

**HTTP 实现：**
- 使用 `WiFiClient` + `client.print()` 手动构造 HTTP POST 请求
- 目标：绕过 UNO R4 WiFi 的 HTTPS 兼容性问题（改用 HTTP）
- 云端 Base URL：`http://cloud1-d4gqmimmo05b12c94-1446329561.ap-shanghai.app.tcloudbase.com`

**NTP 时间同步：**
- 固件不再上传 `ts` 字段，云端使用服务器时间存储
- NTP 同步仅用于固件本地日志时间戳

### 2.6 BLE 配网: BleConfigServer

| 参数 | 值 |
|------|-----|
| Service UUID | 19B10000-e8f2-537e-4f6c-d104768a1214 |
| SSID Char | 19B10001（Write/Notify, 32B） |
| Device ID Char | 19B10002（Read/Notify, 32B） |
| IP Address Char | 19B10003（Read/Notify, 128B） |
| Password Char | 19B10004（Write/Notify, 64B） |

**配网流程：**
1. 固件上电 → BLE 广播（名称 "sEMG_xxxx"）
2. 小程序扫描连接 → 发送 SSID(19B10001) + Password(19B10004)
3. 固件收到 → 保存到 EEPROM → 切换 WiFi STA → 连接路由器
4. DHCP 获取 IP → 更新 IP Char(19B10003)
5. 配网完成 → 固件自动连接 WiFi

**BLE 广播重启（v3.9.39）：**
WiFi 连接成功后，`resumeAdvertising()` 会先 `BLE.stopAdvertise()` 再 `BLE.advertise()`，确保广播干净重启，避免 ArduinoBLE 库在已广播状态下重复调用 `advertise()` 导致广播被意外停止。

### 2.7 业务调度: AppController

**tick() 调度（每 100ms 执行一次）：**

```
1. signalProc.update() → 获取最新 RMS/MDF/Fatigue/Activation/Quality
2. 校准阶段处理：
   CALIB_RELAX:
     - RMS+MDF 累积（含 min/max 追踪）
     - 10秒到期 → 去极值均值 → setRelaxBaseline() → 推送 relax_done
   CALIB_ACTIVE:
     - 峰值 RMS 保持
     - recordCalibMdf() 记录每帧 MDF
     - 15秒到期 → finalizeCalibMdf() → setActiveReference() → 推送 active_done
3. 实时数据广播（RUNNING 态 → sendData）
```

**校准流程（handleRecordRelax / handleRecordActive）：**

| 步骤 | 命令 | 固件行为 | 推送 |
|------|------|---------|------|
| 1 | record_relax | 启动 10 秒 RMS+MDF 累积 | 立即 ACK |
| — | （10秒后） | 去极值均值 → setRelaxBaseline | relax_done |
| 2 | record_active | 启动 15 秒峰值保持 + MDF 缓冲 | 立即 ACK |
| — | （15秒后） | finalizeCalibMdf → setActiveReference | active_done |
| 3 | save_calib | SetUserProfile → UpdatePersonalCalib → 写入 EEPROM | calib_saved |

**save_calib 两种模式：**
- 带 `user_score`（1-10）：完整保存 = profile + calib 一起写入 EEPROM
- 不带 `user_score`：profile-only = 仅将个人信息持久化到 EEPROM（校准数据不变）

**个人信息持久化流程：**
1. 用户填写 profile 表单 → 小程序立即发送 `save_calib{name,age,gender,handedness}`（无 user_score）
2. 固件 `SetUserProfile()` → 个人信息写入 EEPROM
3. 小程序 localStorage 同步更新（作为快照缓存，EEPROM 为权威数据源）

## 3. 小程序端详解

### 3.1 入口: app.js

**启动流程（v10.5）：**
1. `onLaunch`: 初始化全局日志拦截 + 注入监听器（**不**发起连接）
2. `onShow`: 触发统一连接流程（reset → 延迟3s → _connectDevice(true)）
3. 优先通过 BLE 扫描获取设备最新 IP
4. BLE 失败 → 降级使用缓存 IP（wx.storage device_ip）
5. 连接前进行**网络预热**：通过 `wx.request` 发 HTTP 请求到目标 IP 激活手机 TCP 路由表（v10.8）
6. `_connectGen` 代数计数器防止过期 BLE 超时闭包逃逸（v10.5）
7. 监听 BLE IP 回调 → 自动 TCP 连接
8. 监听 TCP 状态变化 → 掉线 Toast 提示 + 自动重连

**连接管理（集中化）（v10.4）：**
- **仅 app.js 调用 `wifiClient.connect()`**，所有页面只注册监听器
- 页面 onShow 不自行发起连接，由 app.onShow 统一管理
- 避免网络请求冲突，使用队列管理重连

**onShow 重连流程（v10.3）：**
```
wifiClient.reset()           ← 取消所有旧重连定时器，挂起重连
→ 3 秒延迟                   ← 等待手机 WiFi 路由恢复
→ _connectDevice(true)       ← force=true 跳过防抖
→ BLE 扫描 → 拿到 IP
→ HTTP 预热（v10.8）         ← 强制内核建立 ARP + 路由
→ 云端连接
```

**全局数据模式：**
- `app.dataMode`: 'idle' | 'streaming'
- 页面切换时自动停止数据流（setDataMode）

### 3.2 云端通信: 云函数调用

**架构：** 小程序不直接连接固件，而是通过微信云函数读取固件上传到云端的数据。

### 3.4 本地存储: storage.js

**单用户设计。localStorage 作为快照缓存（快速 UI 展示），固件 EEPROM 为唯一权威数据源。**

| Key | 内容 | 权威源 |
|-----|------|--------|
| current_user | 当前用户对象 {name, gender, age, handedness, relax_rms, active_rms} | EEPROM |
| user_profile | 用户 profile 备份（与 current_user 同步） | EEPROM |
| calib_data | 校准结果缓存 {relax_rms, active_rms} | EEPROM |
| device_ip | 设备 IP 地址 | BLE 回传 |
| cloud_url | 云函数 Base URL | 本地配置 |
| device_ssid | 已连接 WiFi SSID | BLE 回传 |

**数据流向：**
```
用户输入 → 小程序 localStorage (快照缓存)
         → save_calib{profile} → 固件 EEPROM (权威源)
         
页面加载 ← 小程序 localStorage (快速读，不查固件)
         (如缓存丢失，需重新配网+校准恢复)
```

**API：**
- getCurrentUser() — 读取 current_user
- setCurrentUser(user) — 写入 current_user
- saveCurrentUser(user) — 同时写入 user_profile + current_user
- loadUserProfile() — 读取 user_profile
- clearUserProfile() — 清除所有用户数据

## 4. 页面详细设计
